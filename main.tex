\begin{abstract}
\noindent
We formulate a static multi-trip routing and inter-trip charging problem for an Electrified-Autonomous Flexible Transit (E-AFT) fleet and solve it as a single mixed-integer program with explicit multi-station choice. The bilinear charging relation \(E_{ksr}=p_{ksr}\theta_{ksr}\) is replaced by a station-specific piecewise-linear map from charging time to charged energy, so the integrated model stays linear without sacrificing the tapering shape that drives the operational difference. On a 7-request single-bus deep-recharge instance the linear charging assumption commits to 35.81 kWh of inter-trip recharging in 24.47 minutes; replaying the same charge-time bookings against the nonlinear curve delivers only 31.55 kWh -- a 4.26 kWh shortfall that propagates through battery propagation and renders the discrete plan \emph{infeasible}: linear reaches an objective of 165.51 with five served requests, but the best honest nonlinear plan serves only four at 109.07. The 56.44 gap is therefore not a tuning artefact but a regime where the linear approximation produces an unexecutable plan. The same pathology shows up under the multi-bus configuration (\(|K|=2,\,R=2,\,|Q|=8\)) at a smaller magnitude (gap 6.24, again with an infeasible linear plan). We position this as a state-dependent boundary: the simplified charging assumption is safe when capacity is comfortable relative to demand, and misleading when it is tight enough to push an inter-trip recharge into the slow tail of the taper.
\end{abstract}

\section{Problem Description}

Electrified-Autonomous Flexible Transit (E-AFT) is an emerging and promising transportation mode for passenger travel. It allows passengers to submit travel requests online before traveling. They can detail their travel origins, destinations, and acceptable pickup time. Based on passenger requests, E-AFT routes are generated by selecting the appropriate stops to pick up and drop off passengers. Compared to traditional electric buses, E-AFT has a limited battery capacity that constrains traveling range. The smaller battery capacities make them more sensitive to routing planning to serve passenger requests, which requires further research on the optimization of transportation routing planning. In addition, the majority of current research assumes charging only happens at fixed depots \cite{gao2025integrated}. Based on these research gaps, Jiang et al. (2026) \cite{jiang2026two} developed a two-stage optimization approach for dynamic routing and charging scheduling. They assume that charging is restricted to happen between trips where no passengers are on board. They also assume a linear charging speed for vehicle charging. However, the charging speed can be nonlinear in reality \cite{yi2025charging, montoya2017electric}. Building on \cite{jiang2026two}, this project explores a static version of the trip routing and charging scheduling problem with a nonlinear charging profile and an explicit choice among multiple charging stations. The setting is illustrated in Figure~\ref{fig:concept}. Each bus moves from one stop to another to pick up and drop off passengers, which forms a trip. After all passengers drop off, the bus moves to a charging station to recharge. It then moves to the first bus stop of the next trip. The objective is to decide which passenger requests will be served by each bus, along with the route each bus follows within energy-consumption constraints. The nonlinear aspect of the project arises from the charging process, where the charging rate varies with the battery state instead of remaining constant. As a result, routing, timing, and charging decisions are tightly coupled.

\section{Mathematical Formulation}

Mathematical notation is summarized in Table~\ref{tab:notation}. Compared with the current study \cite{jiang2026two}, this project focuses on a static version in which routing and charging decisions are optimized once for the entire planning horizon. Instead of solving the problem in two heuristic stages, we formulate it in one optimization problem. In addition, this project examines the situation where the charging speed is nonlinear. For simplicity, several components of the original setting are omitted.

The objective function is the operational profit. It includes the revenue from served passenger requests, the fixed vehicle deployment cost, the penalty cost for unserved requests, the routing cost, and the charging cost.

Constraints (2) and (3) describe whether each request is served by a vehicle or not served. Constraints (4) and (5) ensure that the trip is formed, as at least one request is served during the trip. A subsequent trip can only be formed after the current trip is formed. Constraints (6)--(9) ensure that the pick-up and drop-off nodes are visited exactly once if the request is assigned to the trip. Constraint (10) ensures the flow conservation inside each trip. Constraints (11)--(14) define the first and the last node visited in trips by node connections. When there is no incoming arc for a node that is in the trip, it is the first node visited in the trip. If there is no outgoing arc for a node that is in the trip, it is the last node visited in the trip. Constraints (15)--(18) describe time constraints. Constraints (15) and (16) ensure that the arrival time at the pick-up node is within the acceptable time range. Constraint (17) ensures the drop-off time is later than the pick-up time. Constraint (18) describes within-trip time propagation. Constraints (19)--(21) represent the passenger-load changes as the bus picks up and drops off passengers along the way. Constraints (22)--(25) enforce the trip start and end time to equal the arrival time of the first and the last node of the trip. Constraint (26) ensures the arrival time of the starting node of the next trip is later than the arrival time of the last node of the current trip, plus the time traveling between nodes and a charging station, and the charging duration. Constraints (27) and (28) represent the change in battery level. Constraints (29)--(33) represent trip-level battery boundary constraints. Constraint (34) ensures that charging happens only between trips. Constraint (35) presents the relationship between charging power and charging time. Constraints (36) and (37) define the battery-level transitions between trips. Constraints (38)--(40) apply battery boundaries to the battery level along the trip. Constraints (41)--(51) pertain to the binary or non-negative variables. Among these constraints, (35) is a bilinear form that is naturally nonlinear, and the charging power \(p_{ksr}\) can itself be nonlinear.

\section{Implementation and Simplifications}

The model is implemented in Python with Gurobi.

\subsection{Why a Piecewise-Linear Charging Map Instead of the Bilinear Form}

Constraint (35), \(E_{ksr}=p_{ksr}\theta_{ksr}\), is bilinear in two continuous decision variables. Keeping this relation in its bilinear form inside the integrated model forces the solver into a mixed-integer quadratically constrained program (MIQCP) whose spatial branching interacts with the already large binary route-and-assignment space; in a size-limited license regime such as ours, this combination does not scale past two to three requests.

A more conservative relaxation -- McCormick envelopes -- would keep the problem MILP but yield only a convex outer approximation whose gap on tapering charging curves is driven by the largest product term, so solution quality is sensitive to the tightness of the battery-capacity bounds and the charging-time bounds used to build the envelope.

The formulation we adopt takes a different route. Physically, the quantity \(p_{ksr}\) is not an independent control; for a given battery and charger, the charging speed depends on the current state of charge by the battery management system. The published fast-charge profiles for lithium-ion EV batteries follow a constant current (CC) - constant voltage (CV) pattern whose energy-versus-time curve can be well approximated by three to four affine segments \cite{yi2025charging, montoya2017electric}. Similarly, out study captures the nonlinearity of the charging process through a piecewise linear approximation. In the implementation, we keep the \emph{physically observable} quantity, the total energy \(E_{ksr}\) acquired as a function of time \(\theta_{ksr}\), and drops \(p_{ksr}\) entirely as a redundant intermediate variable. The resulting relation is encoded as \(\mathrm{PWL}\bigl(\theta_{ksr}; \theta^{(1)}_s,\ldots,\theta^{(P)}_s; E^{(1)}_s,\ldots,E^{(P)}_s\bigr)\) via Gurobi's \texttt{addGenConstrPWL}. These points $((\theta^{(1)}_s, E^{(1)}_s), \ldots, (\theta^{(P)}_s, E^{(P)}_s))$ are breakpoints of piecewise linear functions. 

This project considers two types of charging processes. The first one is a linear charging that corresponds to evenly spaced breakpoints with a constant slope. The second type is a nonlinear (tapering) charging that is represented by decreasing-slope breakpoints. We also consider different charging speeds of charging stations. This difference is implemented by scaling the time axis of the breakpoints. Figure~\ref{fig:charging-profiles} shows both base charging curves and the station-specific variants. It shows that the charging hub/station 1 has a faster speed of charging than the charging hub 2.

\section{Computational Study}

The first implementation instance uses one bus, three trip slots, two charging stations, and seven passenger requests that are generated randomly on a small synthetic grid. We add a multi-bus instance (two buses, two trip slots, eight requests) to check whether the conclusions sustain for a changed fleet setting. The multi-bus instance is described in Section~\ref{sec:multibus}. Parameters are set as: fixed deployment cost as \$18/bus, travel cost as \$1.1/minute, unserved request penalty cost as \$18/request, charging cost as \$0.55/kWh, bus capacity as 3 passengers, maximum battery capacity as 36 kWh, minimum allowed battery as 4 kWh, and initial battery level as 22 kWh. 

We consider three scenarios where the time windows of pickup times of some requests are different, with all other settings the same. These scenarios are:
\begin{itemize}
    \item \textbf{Baseline}: all seven windows are loose ($\approx 55$ minutes wide). Both charging assumptions are expected to serve all seven requests.
    \item \textbf{Partial recharge}: requests \(r_3\) and \(r_4\) have tightened to roughly 10-minute windows, and their earliest pickup times are set earlier. We consider this scenario because we want to test whether the bus would take these requests on one trip or only serve one request. This scenario involves a moderate charging at stations, so it is named as ``partial recharge''.
    \item \textbf{Deep recharge}: the earliest pickup times and time windows of \(r_3\), \(r_4\), and \(r_5\) are set closer so that these requests come back-to-back. This scenario involves serving all of them in one or two trips and requires much energy between trips, so it is named as ``deep recharge''.
\end{itemize}

In the \textbf{baseline} case under two charging functions, the bus serves all seven requests at the same objective of \$353.64. The energy acquired is identical (35.95 kWh), but the nonlinear curve takes 41.27 minutes while the linear curve takes 24.57 minutes. It might be because this case requires a charging time that stretches into the slow tail of the nonlinear charging curve and requires more time than using a linear charging curve. This finding suggests that the tapering charging is not uniformly faster than constant-rate charging. Tapering charging is faster only in the early segment when the charging time is relatively small.

\subsection{Problem Size and Runtime}

Model size is dominated by the arc variables \(x_{kijr}\) and the first/last node indicators, which scale as \(|K|\,|N|^{2}|R|\) with \(|N|=2|Q|\) service nodes. With Gurobi 11 default settings, the single-bus configuration takes 1.8 seconds to solve for the baseline, 3.3-3.9 seconds for partial recharge, and 2.3-2.7 seconds for deep recharge. The linear and nonlinear modes share the same model size because each station carries the same number of breakpoints. Their run times are similar.

\subsection{Multi-Bus Robustness Check}
\label{sec:multibus}

In the above model settings, we only consider one bus in the configuration. A natural concern is what solutions would be if there are multiple buses. We therefore re-run the deep-recharge scenario with two homogeneous buses, two trip slots per bus, and eight requests (instances were generated with the same seed, but with the additional bus and request added).

Results show that under the linear charging assumption, the model reaches an optimal value of \$300.22 by serving seven of the eight requests. One bus receives an 18.18 kWh inter-trip recharge to conduct a second trip for three requests. Under the nonlinear curve, the optimal solution serves seven requests with an optimal value of \$293.98. The cross-evaluation checks how the plan under a linear charging curve performs regarding charging under the nonlinear curve. It shows similar results that the plan under a linear curve is infeasible inside the same inter-trip window under the nonlinear curve. Re-optimization under a nonlinear curve spreads load across two buses with smaller per-trip recharges. The optimal plan serves seven requests with an optimal value of \$293.98. Compared to the single-bus configuration, the run times of the multi-bus configuration grow by roughly 1.6\(\times\) in arc variables and take 14-59 seconds to solve, depending on the scenarios.

In both the single-bus (\(|K|=1,\,|R|=3\)) and the two-bus (\(|K|=2,\,|R|=2\)) configurations, the optimal values under the nonlinear curve are lower than the optimal values under the linear curve. It might be because the capacity in terms of available buses and trip slots is limited. It forces a substantial inter-trip recharge into the slow tail of the nonlinear charging curve. We expect that further expanding the capacity, such as adding more buses, more trip slots, or both, would bring a change. In this scenario, each trip can become shorter, and each inter-trip recharge can stay inside the early segment of the charging curves, where the charging speed of the nonlinear charging curve is faster than that of a linear charging curve.

We expect that further loosening capacity (more buses, more trip slots, or both) would dilute this regime, since each trip becomes shorter and each inter-trip recharge stays inside the early segment where the two charging assumptions barely disagree -- but we have not run a saved sweep over those configurations and do not claim a quantitative boundary here.



\section{Limitations}

The bilinear form \(E_{ksr}=p_{ksr}\theta_{ksr}\) in constraint (35) is replaced by a piecewise-linear map from charging time to charged energy. This keeps the integrated formulation a MIP rather than a MIQCP and is what makes the deep-recharge experiment tractable on a laptop, but it does mean the implementation no longer carries the original bilinear relation exactly. The test instances are synthetic, generated on a small Euclidean grid, and we do not benchmark against published transit datasets. The model is also static; it does not handle dynamic request arrivals or stochastic travel times. Finally, the symmetry-breaking we apply across identical buses is a lex-order constraint on bus usage, which is sound but loose; a tighter symmetry break would likely accelerate the multi-bus experiments further.

\section{Conclusion}

We formulate the static E-AFT routing and inter-trip charging problem as a single MIP and replace the bilinear charging relation by a station-specific piecewise-linear map guarded by indicator constraints. On a 7-request single-bus instance, the linear and nonlinear charging assumptions agree in the baseline and partial-recharge regimes, and differ only in charging time. The deep-recharge regime is where they come apart: linear commits to 35.81 kWh of inter-trip recharging that the nonlinear curve, in the same booked time, only supplies as 31.55 kWh -- a 4.26 kWh shortfall that breaks the discrete plan's feasibility check. The honest nonlinear plan instead serves four requests at 109.07. This 56.44 gap is the central result -- it is not a tuning artefact, it measures the cost of relying on a charging approximation that the physical battery cannot match. The same pattern shows up at smaller magnitude in the two-bus, eight-request configuration. We read the takeaway as state-dependent rather than monotone: the linear approximation is safe when capacity is comfortable relative to demand, and misleading when capacity is tight enough to force a deep recharge into the slow tail of the taper.
